# Flight School Admin Portal

This document explains the new Flight School Admin features added to ClearedToGo.

## Overview

The app now supports two user roles:
- **Pilot** - Regular users who use the app for flight planning and checklists
- **Flight School Admin** - Administrators who can view pilot PDFs and customize checklists

## Architecture

### Clean Architecture with Repository Pattern

The implementation follows clean architecture principles to make it easy to swap implementations later:

```
Models → Repositories (interfaces) → Local Implementations
```

**Models** (`lib/models/`):
- `user_role.dart` - User and UserRole enum
- `flight_school.dart` - FlightSchool entity
- `pdf_record.dart` - PDF metadata
- `checklist_models.dart` - ChecklistTemplate, ChecklistSection, ChecklistItem

**Repository Interfaces** (`lib/repositories/`):
- `auth_repository.dart` - Authentication operations
- `pdf_repository.dart` - PDF storage and retrieval
- `checklist_repository.dart` - Checklist management

**Local Implementations** (`lib/repositories/`):
- `local_auth_repository.dart` - Uses SharedPreferences + seeded demo account
- `local_pdf_repository.dart` - Stores PDFs in app documents directory
- `local_checklist_repository.dart` - Stores custom checklists in SharedPreferences

### Replacement Strategy

When ready to integrate with AWS Cognito + S3 + backend:

1. **Authentication**: Create `CognitoAuthRepository` implementing `AuthRepository`
2. **PDFs**: Create `S3PdfRepository` implementing `PdfRepository`
3. **Checklists**: Create `ApiChecklistRepository` implementing `ChecklistRepository`
4. **Swap**: Update dependency injection in services to use new implementations

All UI code remains unchanged - just swap the repository implementations.

### How Login Works

1. User enters email + password on login screen
2. App tries **local auth first** (checks against local user database)
3. If local auth fails, falls back to **Cognito auth** (for existing pilot users)
4. After successful login, user is routed based on role:
   - **Pilot** → Normal home screen
   - **Flight School Admin** → Flight School Dashboard

## Flight School Portal Features

### 1. Dashboard (`/flight-school/dashboard`)

Landing page for admins showing:
- Total PDFs count
- Custom checklists count
- Quick action buttons to navigate to other screens

### 2. PDF Library (`/flight-school/pdfs`)

View all PDFs submitted by pilots linked to the flight school:

**Features**:
- Search by pilot name, aircraft type, or title
- Filter by PDF type (weight & balance, tech log, fuel uplift, etc.)
- View PDF details (pilot, aircraft, date)
- Open PDFs for viewing
- Delete PDFs

**How it works**:
- Every PDF generated by pilots is now automatically saved and tagged with:
  - Pilot name and ID
  - Aircraft type and registration
  - Flight school ID (if pilot is linked to a school)
  - Timestamp
  - PDF type
- Flight school admins see only PDFs from their school

### 3. Checklist Editor (`/flight-school/checklists`)

Customize aircraft checklists for your flight school:

**Features**:
- Select aircraft type (C152, C172, PA28)
- View default checklist or create custom version
- Add/edit/delete sections
- Add/edit/delete checklist items
- Reorder sections by drag-and-drop
- Changes are saved per flight school

**Override Logic**:
- Default checklists are built-in (hardcoded for now)
- Flight schools can create custom versions
- When a pilot linked to a flight school views a checklist, they see:
  1. **Custom checklist** if one exists for their school + aircraft type
  2. **Default checklist** if no custom version exists

## PDF Storage

### How PDFs are Stored

**Current Implementation** (Local):
1. Pilot generates PDF in any screen (weight & balance, tech log, etc.)
2. PDF is saved to temp directory
3. `PdfStorageHelper.savePdf()` is called, which:
   - Copies PDF to app documents directory (`flight_pdfs/`)
   - Extracts user metadata (pilot name, flight school ID)
   - Creates a `PdfRecord` with all metadata
   - Saves record to SharedPreferences
   - Returns managed file path
4. PDF opens for viewing/sharing

**Future Implementation** (Cloud):
Replace `LocalPdfRepository` with `S3PdfRepository`:
1. Upload PDF bytes to S3 bucket
2. Get S3 URL
3. Send metadata + URL to backend API
4. Backend stores in database
5. Flight school admins query backend API for their school's PDFs

### Data Storage Locations

**LocalAuthRepository**:
- `shared_preferences`: `current_user_v2` (current logged in user)
- `shared_preferences`: `users_database` (all local user accounts)

**LocalPdfRepository**:
- File system: `{app_documents}/flight_pdfs/*.pdf`
- `shared_preferences`: `pdf_records` (PDF metadata as JSON array)

**LocalChecklistRepository**:
- `shared_preferences`: `custom_checklists` (JSON array of custom checklists)

## Integration Guide

### To Add PDF Saving to a New Screen

1. Import the helper:
```dart
import 'package:clearedtogo/services/pdf_storage_helper.dart';
```

2. After generating PDF bytes, call:
```dart
final filePath = await PdfStorageHelper().savePdf(
  pdfBytes: pdfBytes,
  title: 'My PDF Title',
  pdfType: 'weight_balance', // or 'tech_log', 'fuel_uplift', etc.
  aircraftType: 'C152', // optional
  aircraftRegistration: 'G-ABCD', // optional
);

OpenFile.open(filePath);
```

### To Use Custom Checklists

1. Inject `ChecklistRepository`:
```dart
final checklistRepo = LocalChecklistRepository();
await checklistRepo.init();
```

2. Load checklist for current user:
```dart
final user = AuthServiceManager().currentUser;
final template = await checklistRepo.getChecklistTemplate(
  aircraftType: 'C152',
  flightSchoolId: user?.flightSchoolId, // null for pilots without school
);

// template will be custom if exists, otherwise default
```

### To Switch to AWS Cognito

1. Create `CognitoAuthRepository`:
```dart
class CognitoAuthRepository implements AuthRepository {
  @override
  Future<User> signIn(String email, String password) async {
    // Use Amplify.Auth.signIn()
    // Map Cognito user to our User model with role
  }
  // ... implement other methods
}
```

2. Update `AuthServiceManager.init()`:
```dart
_authRepository = CognitoAuthRepository(); // instead of LocalAuthRepository()
```

## Migration from Old Auth

The app supports both old (Cognito) and new (repository-based) auth:

1. **Existing users**: Continue to use Cognito auth
2. **New admin users**: Use local auth (demo account)
3. **AuthGate** checks both auth systems
4. **LoginScreen** tries local auth first, falls back to Cognito

This allows gradual migration without breaking existing users.

## Security Notes

**Current Local Implementation**:
- Passwords stored in plain text in SharedPreferences
- **DO NOT use in production**
- Only for demo/development purposes
- Must replace with proper Cognito auth

**Production Requirements**:
- Use AWS Cognito for authentication
- Store PDFs in S3 with proper IAM policies
- Use signed URLs for PDF access
- Implement proper role-based access control
- Never store credentials in SharedPreferences
